Asteroids Requirements and Status
=================================

Purpose
-------
This document tracks the intended gameplay, technical structure, and known gaps for the Asteroids game in this repository. It reflects the current refactored codebase, not just the original reverse-engineering notes.

Status Key
----------
[x] Implemented
[~] Implemented with known deviation or approximation
[ ] Not implemented


Gameplay Requirements
---------------------

Core Controls
-------------
[x] Up arrow applies forward thrust
[x] Left arrow rotates the ship counter-clockwise
[x] Right arrow rotates the ship clockwise
[x] Space fires bullets
[x] P toggles pause during gameplay
[x] Enter starts from attract mode and restarts from game over

Ship Mechanics
--------------
[x] Ship accelerates in the direction it is facing
[x] Ship rotation is continuous while rotate keys are held
[x] Ship movement uses inertia
[~] Ship speed is practically constrained by movement tuning rather than a hard arcade-authentic cap
[x] Ship wraps around screen edges
[x] Ship has safe-spawn handling before play resumes
[x] Ship death is delayed until active threats/effects are resolved

Bullet Mechanics
----------------
[x] Ship bullets spawn from the ship nose
[x] Bullet direction follows firing angle
[x] Bullet velocity inherits source movement
[x] Bullets expire after a limited lifespan
[x] Bullets are removed on collision
[x] Ship bullets and UFO bullets are tracked separately by ownership/team metadata

Asteroid Mechanics
------------------
[x] Asteroids spawn at randomized positions around the playfield
[x] Asteroids move in randomized directions
[x] Asteroids wrap around the screen
[x] Large asteroids split into medium asteroids
[x] Medium asteroids split into small asteroids
[x] Small asteroids are destroyed without further splitting
[x] A new wave starts when all asteroids are cleared
[~] Asteroids do not bounce off each other; collision between asteroids is not part of the current simulation

UFO Mechanics
-------------
[x] UFOs spawn at timed intervals
[x] Only one UFO can be active at a time
[x] UFOs move across the screen with horizontal travel and vertical variation
[x] Large and small UFO variants exist
[x] Large UFO shots are inaccurate/random
[x] Small UFO shots target the player
[x] UFO bullets use a limited lifespan
[x] UFO bullets can persist briefly after UFO destruction
[~] UFOs leave the screen and are destroyed rather than wrapping to the opposite side
[ ] UFO frequency should increase with progression to better match the arcade game
[ ] UFO behavior should scale with progression, especially small-UFO threat level

Collision Rules
---------------
[x] Ship collides with asteroids
[x] Ship collides with UFOs
[x] Ship can be hit by UFO bullets
[x] Ship can be hit by non-owner player bullets
[x] Player bullets destroy asteroids
[x] Player bullets destroy UFOs
[x] UFO bullets can destroy asteroids
[x] Collision resolution is handled by a dedicated collision system and hit resolver

Progression and Scoring
-----------------------
[x] The player has a limited number of lives
[x] Supports one-player and two-player selection flow
[x] Scores are tracked per player
[x] Score is awarded for asteroid destruction by size
[x] Score is awarded for UFO destruction
[x] Player switching occurs after death when appropriate
[x] Game over transitions back to attract mode
[x] High score is persisted via cookie storage
[ ] Difficulty scaling should more closely match the arcade game
[ ] Later waves should increase pressure through stronger asteroid density and/or pace
[ ] Small UFO accuracy should increase as progression advances
[ ] UFO spawn timing should become more aggressive as progression advances
[ ] Higher progression should create a noticeably more dangerous playfield than the opening wave

Visual and Audio Requirements
-----------------------------
[x] Vector-style rendering for the ship, asteroids, and UFOs
[x] HUD shows player scores, lives, and high score
[x] Explosions play for ship, asteroid, and UFO destruction
[x] Ship thrust sound plays during thrust
[x] Bullet fire sound plays for the player
[x] Asteroid hit sounds vary by asteroid size
[x] UFO looping sound plays while the UFO is active
[x] UFO looping sound stops when the UFO is destroyed or exits the screen
[~] No separate background music system is used; the game relies on arcade-style effects instead
[ ] Screen shake / hit flash effects are not currently implemented


Architecture Requirements
-------------------------

Top-Level Layout
----------------
[x] `game.js` is the top-level game entry point and state handler
[x] `global.js` holds Asteroids-specific configuration/constants
[x] Root-level actor files remain easy to find:
    `ship.js`, `ufo.js`, `asteroid.js`

Folder Organization
-------------------
[x] `ui/` contains presentation helpers and attract-mode scene files
[x] `runtime/` contains app/session/runtime coordination
[x] `state/` contains state machine and persistent score storage
[x] `systems/` contains gameplay systems and hit/collision/score orchestration
[x] `combat/` contains bullet and bullet-factory logic
[x] `world/` contains world and manager ownership for asteroid/UFO simulation

Responsibility Boundaries
-------------------------
[x] `Ship` no longer owns world managers
[x] World ownership is centralized in `world/world.js`
[x] Collision resolution is separated from the world coordinator
[x] Hit effects are separated from collision detection
[x] Bullet spawn policy is separated from bullet storage
[x] Score accumulation is separated from the world update loop
[x] Session/player bookkeeping is separated from top-level game state
[x] HUD and screen presentation are separated from active gameplay runtime
[x] State transitions are centralized in the Asteroids state machine
