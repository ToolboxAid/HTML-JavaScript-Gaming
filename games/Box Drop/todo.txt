Box Drop engine upgrade plan

Goal:
Keep Box Drop as a compact engine game while aligning it with the current runtime conventions and making it a cleaner small-project example.

Project-specific checklist:
[ ] Review `GameBase`, `onInitialize(runtimeContext)`, and `gameLoop(deltaTime, runtimeContext)` against current engine conventions
[ ] Remove avoidable startup/debug logging noise
[ ] Keep attract, player select, game, and game-over flow clear and easy to follow
[ ] Review whether player/enemy logic should remain in `game.js` or move into helper files

Shared engine alignment reference:

Target project structure for both `samples/` and `games/`:

```text
Project Name/
  index.html
  styles.css               # optional, only when page-level styling is needed
  global.js
  game.js
  data/                    # optional structured content/config data
  objects/                 # optional project-specific object/entity classes
  systems/                 # optional project-specific systems/managers
  state/                   # optional state machines, stores, and transition helpers
  runtime/                 # optional app/session/bootstrap helpers for larger games
  world/                   # optional world orchestration, spawning, collision groups
  combat/                  # optional weapons, bullets, attack logic
  scenes/                  # optional scene/state modules for larger projects
  ui/                      # optional HUD/menu/canvas UI helpers
  assets/                  # optional images, audio, maps, sprite sheets, etc.
  README.md                # optional per-project notes when the project teaches something specific
  todo.txt                 # optional local upgrade/review plan
```

Directory structure guidelines:

[ ] Keep `index.html` minimal and focused on bootstrapping the module entry point
[ ] Keep `game.js` as the runtime shell: setup, owned input, state flow, update, and render orchestration
[ ] Keep `global.js` for shared project config only
[ ] Put raw content and definitions in `data/` instead of mixing large config blocks into `game.js`
[ ] Put project-specific entities in `objects/`
[ ] Put project-specific coordination logic in `systems/` only when it improves clarity
[ ] Use `state/` when state transitions, save data, or high-score stores deserve their own boundary
[ ] Use `runtime/` when app/session/bootstrap concerns should be separated from the main game shell
[ ] Use `world/` when gameplay objects, spawning, and world-level coordination need their own layer
[ ] Use `combat/` when weapons, bullets, or attack interactions become large enough to separate cleanly
[ ] Add `scenes/` only when the project has enough state complexity to justify scene modules
[ ] Keep UI and HUD helpers under `ui/` when canvas presentation becomes large enough to separate
[ ] Use `assets/` for project-owned resources instead of scattering them across the root folder
[ ] Avoid adding folders that do not teach or simplify anything for small samples

Ideas taken from the current `games/` directory:

[ ] `games/Asteroids/` shows good larger-project boundaries with `runtime/`, `state/`, `systems/`, `ui/`, `world/`, and `combat/`
[ ] `games/Frogger/` shows the value of dedicated object folders plus content/config helpers such as `gameObjects/`, `levelData.js`, and `levelManager.js`
[ ] Smaller games such as `Pong Game/`, `Tic-Tac-Toe/`, and `Connect 4/` show that not every project needs subfolders if a flat structure is still readable
